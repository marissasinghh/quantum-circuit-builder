Simulate API: JSON → validation → DTOs → Cirq → response

This page shows how a POST /api/simulate body travels through the backend. The level-definition shape (GateStep / steps[]) lives outside this HTTP flow—it is used for the built-in target library, not for the trial circuit JSON.

Example request body (includes theta on one gate) 
{
  "number_of_qubits": 2,
  "gates": ["H", { "gate": "RX", "theta": 1.2 }, "CNOT"],
  "qubit_order": [[0], [1], [0, 1]],
  "target_unitary": "SWAP"
}
Row i: gates[i] pairs with qubit_order[i] (H on q0, RX(1.2) on q1, CNOT 0→1).

Flowchart + level definition (sidebar)

Main path: trial circuit from HTTP

Files: app/api/simulate.py → app/utils/unitary_payload.py → app/dto/unitary.py → app/controllers/simulate.py → app/services/circuit_builder.py → app/config/gates.py → app/services/simulator.py.

Level definition — not in this request

Canonical targets are authored as Python data with self-contained steps. Each step includes order on the same object (no parallel qubit_order array).

app/utils/types.py — GateStep (TypedDict)

  • gate (string)
  • order (list of qubit indices)
  • theta optional — same meaning as in the HTTP dict gate

app/config/target_library.py

  • TARGET_LIBRARY[name] has steps, num_qubits, expected_outputs, …

app/services/target_builder.py

  • TargetUnitaryBuilder.build(name, qubits) loops steps, reads gate, order, theta, calls CirqGateMapper.apply (same mapper as the trial path).

app/utils/helpers.py

  • build_target_truth_table / related helpers use TARGET_LIBRARY when not building the circuit from scratch.
Where it “shows up” in the simulate flow
The client sends target_unitary: "SWAP" (a string key).
If validate_target is on, the backend builds the target circuit from TARGET_LIBRARY["SWAP"].steps — that is the level-definition shape.
The trial circuit never uses steps[]; it uses gates + qubit_order only.

What each DTO holds (and how it changes)

UnitaryDTO

app/dto/unitary.py
number_of_qubits
Positive integer — how many line qubits to allocate.
gates
List parallel to qubit_order. Each entry: a string (e.g. "CNOT") or a dict {"gate": "RX", "theta": 1.2}. Nothing is “split out” later — the lists are passed through as-is.
qubit_order
gates[i] is applied using qubit_order[i] (wiring for that row).

Not included: target_unitary from JSON is a separate string; it is not a field on this DTO.

TruthTableDTO

app/dto/truth_table.py
Structure
input: list[str], output: list[str] — parallel columns (same length when filled).
Created empty: TruthTableDTO([], []) for trial and for target at the start of simulate_unitaries.

Filled during simulation: each basis state run appends one row via CircuitSimulator.simulate_and_update (inputs like |01⟩, outputs from Cirq statevector notation).

Serialized for HTTP: .to_dict() → { "input": [...], "output": [...] }.

Validation (no DTO)

app/utils/unitary_payload.py
validate_simulate_unitary_json
Checks the raw dict: required keys, list lengths, each gate string or {"gate", "theta"?}, numeric theta. Raises ValueError → API returns ResponseBuilder.fail. Does not construct a dataclass.

Response payload (dict, not a DTO class)

controllers/simulate.py return value
Contents
Message, trial_truth_table and target_truth_table as dicts from TruthTableDTO.to_dict(), plus validation_mode.
